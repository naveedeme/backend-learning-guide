Learning Philosophy

The 70/30 rule: Spend 70% of your time building real projects and 30% studying concepts. Reading about databases without building with them is almost useless. Every section includes a project to build.
Depth over breadth: Master PostgreSQL deeply before touching 5 databases shallowly. Companies care that you can diagnose a slow query, not that you've "used" 10 different systems.

Core Concept

SQL vs NoSQL

The most important architectural decision in backend development. Understanding this deeply separates junior from senior engineers.

The Simple Mental Model

SQL (Relational) — Think of a perfectly organized Excel spreadsheet. Data lives in tables with rows and columns. Relationships between tables are defined upfront. Great when your data has a clear, consistent shape.

NoSQL (Non-Relational) — Think of a flexible filing cabinet. Each "document" can have a different structure. Great when data shape varies wildly or you need to scale horizontally to millions of users.

Deep Comparison

Dimension SQL (PostgreSQL, MySQL) NoSQL (MongoDB, Redis)
Data structureTables, rows, columns (fixed schema)Documents, key-value, graphs (flexible)
SchemaDefined upfront, strictFlexible, can evolve per document
RelationshipsJOINs, foreign keys, enforced by DBEmbedding or manual references in app code
ACID compliance✓ Full — transactions guaranteed⚠ Varies — MongoDB has it, Redis does not
ScalingVertical scaling (bigger server) primarilyHorizontal scaling (more servers) naturally
Query languageSQL — universal, powerful, transferableDB-specific APIs (less transferable)
Schema changesPainful (migrations required)Easy (just add new fields)

When To Choose SQL

  • Financial transactions (money must not be lost or duplicated)
  • Complex relationships between entities (users, orders, products, payments)
  • Compliance-heavy industries (healthcare, banking, legal)
  • You need complex queries across multiple tables
  • Your data shape is well-understood upfront
  • Team is comfortable with SQL (don't switch just to switch)

When To Choose NoSQL

  • Truly variable document structure (product catalog with different attributes per category)
  • Massive write throughput (millions of events per second)
  • Horizontal scaling is a non-negotiable requirement
  • Caching, sessions, leaderboards (Redis)
  • Real-time collaborative features
  • Rapid prototyping where schema is unknown

The Real-World Answer (2026)

Most startups and mid-size companies use PostgreSQL for 80%+ of their data — even "big tech" companies. MongoDB peaked in hype around 2015–2018. PostgreSQL's JSONB column means you get relational AND flexible schema in one database. The industry has largely converged on: PostgreSQL as primary, Redis for caching/queues, and occasionally MongoDB for specific document use cases.

🪶 SQLite — The Humble Giant

Note: SQLite is NOT for concurrent web applications — it doesn't handle multiple simultaneous writers well. But it's perfect for learning SQL before you move to PostgreSQL.

SQL Fundamentals — Learn This Order

Week 1–2

Basic Queries

-- SELECT basics SELECT id, name, email FROM users WHERE active = true; -- Aggregations SELECT department, COUNT(*) AS employee_count, AVG(salary) AS avg_salary FROM employees GROUP BY department HAVING COUNT(*) > 5 ORDER BY avg_salary DESC; -- JOINs SELECT u.name, o.total, o.created_at FROM users u INNER JOIN orders o ON u.id = o.user_id WHERE o.status = 'completed' ORDER BY o.created_at DESC;
Week 3–4

CTEs and Window Functions

-- CTE (Common Table Expression) WITH recent_orders AS ( SELECT user_id, SUM(total) AS total_spent FROM orders WHERE created_at > NOW() - INTERVAL '30 days' GROUP BY user_id ) SELECT u.name, ro.total_spent FROM users u JOIN recent_orders ro ON u.id = ro.user_id ORDER BY ro.total_spent DESC LIMIT 10; -- Window function — running total SELECT date, revenue, SUM(revenue) OVER (ORDER BY date) AS running_total FROM daily_sales;

Database Design — Normalization

Normalization is the process of organizing your database to reduce redundancy and improve data integrity. Think of it as keeping data in exactly one place.

1NF — First Normal Form

Each column holds atomic (single) values. No arrays in columns. Every row is unique.

2NF — Second Normal Form

Must be 1NF. Every non-key column depends on the WHOLE primary key (eliminates partial dependencies).

3NF — Third Normal Form

Must be 2NF. Non-key columns must NOT depend on other non-key columns (eliminates transitive dependencies).

Rule of thumb: Normalize to 3NF, then selectively denormalize for performance. Premature denormalization is one of the most common junior mistakes.

Indexing — The Most Impactful Skill

An index is a separate data structure that makes lookups fast — like a book's index vs reading every page. Without indexes, every query scans the entire table.

-- Create index on frequently queried column CREATE INDEX idx_users_email ON users(email); -- Composite index for multi-column queries CREATE INDEX idx_orders_user_status ON orders(user_id, status); -- Partial index (only index subset of rows) CREATE INDEX idx_active_users ON users(last_login) WHERE active = true; -- Analyze a slow query EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42 AND status = 'pending';
Don't over-index! Every index slows down INSERT/UPDATE/DELETE. Index columns that appear in WHERE clauses and JOIN conditions of frequently run queries.

Transactions — Keeping Data Consistent

A transaction groups multiple operations so they either ALL succeed or ALL fail. Critical for financial operations, inventory management, anything involving money or sensitive data.

-- Transfer money between accounts (ACID example) BEGIN; UPDATE accounts SET balance = balance - 500 WHERE id = 1; -- debit sender UPDATE accounts SET balance = balance + 500 WHERE id = 2; -- credit receiver -- Both succeed → COMMIT, or both fail → ROLLBACK COMMIT;

🍃 MongoDB Deep Dive

Document databases, when to use them, and how to work with them effectively.

Core MongoDB Concepts

Document Model

Data stored as BSON (Binary JSON). Each document can have different fields — no schema enforced at DB level (though you should enforce it in code).

Collections

Groups of documents — equivalent to tables in SQL. A "users" collection holds all user documents.

Aggregation Pipeline

MongoDB's version of complex SQL queries. Chain stages: $match → $group → $sort → $project. Powerful but different from SQL.

Mongoose ODM

The standard Node.js library for MongoDB. Adds schema validation, middleware hooks, and a cleaner API. Always use Mongoose in Node.js projects.

MongoDB with Node.js — Code Pattern

// Mongoose Schema — always define a schema even in NoSQL const userSchema = new Schema({ name: { type: String, required: true }, email: { type: String, required: true, unique: true }, role: { type: String, enum: ['user', 'admin'], default: 'user' }, profile: { // Embedded document bio: String, avatar: String, }, createdAt: { type: Date, default: Date.now } }); // Query with aggregation pipeline const result = await Order.aggregate([ { $match: { status: 'completed' } }, { $group: { _id: '$userId', total: { $sum: '$amount' } } }, { $sort: { total: -1 } }, { $limit: 10 } ]);

⚡ Redis — The Speed Layer

Why every production app needs Redis and how to use it effectively.

Redis Data Structures

Strings

Simple key-value. Most common use: caching API responses, storing session tokens.

Hashes

Like a mini object/record. Use for storing user session data with multiple fields.

Lists

Ordered lists. Use for activity feeds, queues (push/pop from either end).

Sets

Unique values. Use for tracking unique visitors, tags, permissions.

Sorted Sets

Values with scores. Perfect for leaderboards, rate limiting, priority queues.

Pub/Sub

Message broadcasting. Use for real-time notifications between services.

Caching Pattern with Node.js

import { createClient } from 'redis'; const redis = createClient(); async function getUserWithCache(userId) { const cacheKey = `user:${userId}`; // 1. Check cache first (fast — in memory) const cached = await redis.get(cacheKey); if (cached) return JSON.parse(cached); // 2. Cache miss — query database (slow) const user = await db.query('SELECT * FROM users WHERE id=$1', [userId]); // 3. Store in cache for next time (TTL = 1 hour) await redis.setEx(cacheKey, 3600, JSON.stringify(user)); return user; }

Rate Limiting with Redis

async function rateLimiter(req, res, next) { const key = `rate:${req.ip}`; const requests = await redis.incr(key); if (requests === 1) { await redis.expire(key, 60); // 60 second window } if (requests > 100) { return res.status(429).json({ error: 'Too many requests' }); } next(); }

Request Lifecycle

  1. Browser → CDN: Static assets (JS, CSS) served from edge cache (near user)
  2. API Request → Load Balancer: Routes to least-loaded server instance
  3. Middleware: Auth check (verify JWT), rate limit check (Redis), logging
  4. Cache check: Does Redis have this response? Return immediately if yes
  5. Database query: Reads from replica, writes to primary
  6. Cache store: Store result in Redis with TTL
  7. Response: JSON → browser → React updates UI

Avoid These

Common Beginner Mistakes

What separates developers who grow fast from those who stay stuck. Learn from others' mistakes.

Database Mistakes

🚫 Not using parameterized queries (SQL Injection)

// ❌ NEVER do this — SQL injection vulnerability const query = `SELECT * FROM users WHERE email = '${email}'`; // ✅ Always use parameterized queries const query = 'SELECT * FROM users WHERE email = $1'; const result = await db.query(query, [email]);

🚫 Selecting everything with SELECT *

SELECT * returns all columns including ones you don't need — wastes bandwidth and prevents index-only scans. Always select specific columns.

🚫 No indexes on foreign keys

PostgreSQL does NOT automatically index foreign key columns. Every JOIN query on a foreign key without an index does a full table scan.

-- Add indexes on FK columns you JOIN on CREATE INDEX idx_orders_user_id ON orders(user_id); CREATE INDEX idx_comments_post_id ON comments(post_id);

Architecture Mistakes

🚫 Storing passwords in plain text

Always hash passwords with bcrypt (cost factor 12+). Never log passwords. Never store them in plaintext or with reversible encryption.

🚫 Putting business logic in the database

Stored procedures and triggers make your app tightly coupled to the database. Keep business logic in application code where it's testable and version-controlled.

🚫 Not handling database errors

Unique constraint violations, foreign key violations, and deadlocks WILL happen in production. Your API must catch and handle them gracefully instead of crashing.

🚫 The N+1 query problem

// ❌ N+1 — 1 query for posts + N queries for each author const posts = await db.query('SELECT * FROM posts'); for (const post of posts) { post.author = await db.query('SELECT * FROM users WHERE id=$1', [post.user_id]); } // ✅ 1 query with JOIN const posts = await db.query(` SELECT posts.*, users.name AS author_name, users.avatar FROM posts JOIN users ON posts.user_id = users.id `);

Learning Mistakes

🚫 Tutorial hell — watching without building

Watching 50 hours of tutorials without building a single original project. After each tutorial, immediately build something DIFFERENT using the same concepts.

🚫 Skipping SQL to learn MongoDB first

SQL teaches you to think about data relationships. MongoDB lets you skip that thinking — and that's why many MongoDB applications have terrible data models. Learn SQL first, always.

🚫 Not reading error messages

Most beginners see a red error and immediately Google it or paste it into ChatGPT. First: read the full error. Then: check the stack trace. Then: search. You'll solve 50% of problems yourself by just reading carefully.
